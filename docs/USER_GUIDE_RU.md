## 1. Назначение приложения

`WorkplaceMonitor` — Android-приложение для мониторинга параметров рабочего места и состояния работника в реальном времени.

Приложение:
- считывает данные со встроенных датчиков смартфона;
- формирует телеметрию;
- передает телеметрию по MQTT (например, в Yandex IoT Core);
- поддерживает интеграцию с внешними сервисами уведомлений (боты, облачные функции, скрипты).

### Какие задачи решает приложение
- контроль освещенности рабочего места;
- оценка активности работника по шагомеру;
- оценка вибрации/резких движений по акселерометру;
- оценка уровня шума по пиковому значению микрофона;
- передача данных в аналитический контур для построения предупреждений.

**Важно:** приложение не является медицинским устройством.  
Оно предназначено для инженерного мониторинга условий работы, а не для диагностики здоровья.

## 2. Системные требования

### 2.1 Минимальные требования
- Android **9.0+** (API 28+);
- доступ к интернету;
- доступ к MQTT-брокеру.

### 2.2 Датчики

| Датчик | Обозначение в payload | Обязательность |
|---|---|---|
| Акселерометр (`TYPE_ACCELEROMETER`) | `X`, `Y`, `Z` | Желательно |
| Освещенность (`TYPE_LIGHT`) | `L` | Желательно |
| Шагомер (`TYPE_STEP_COUNTER`) | `S` | Опционально (может отсутствовать) |
| Микрофон (`MediaRecorder maxAmplitude`) | `M` | Опционально |

### 2.3 Разрешения
- `RECORD_AUDIO` — для микрофона;
- `ACTIVITY_RECOGNITION` — для шагомера (особенно на Android 10+).

**Примечание:** если датчик недоступен или разрешение не выдано, приложение продолжает работу, но исключает соответствующее поле из отправки.

## 3. Установка и первый запуск

### 3.1 Установка
1. Установите APK на устройство:
   - через Android Studio (`Run app`), или
   - через ADB (`adb install`), или
   - вручную из файла.
2. Откройте приложение.

### 3.2 Первый запуск
1. Нажмите `Settings`.
2. Заполните MQTT-параметры.
3. Выберите активные датчики.
4. Укажите интервал публикации (`Publish interval (ms)`).
5. Выберите режим уведомлений.
6. Нажмите `Save settings`.
7. Вернитесь на главный экран и нажмите `Start`.

### 3.3 Что проверить после запуска
- данные датчиков на экране обновляются;
- поле `MQTT Payload Preview` меняется;
- в логах/подписчике приходят сообщения.

**Важно:** минимально рекомендуемый интервал отправки — `1000 ms`.

## 4. Настройка подключения к Yandex IoT Core

Откройте экран `Settings` и заполните поля:

| Поле | Что указывать | Пример |
|---|---|---|
| Broker | MQTT endpoint с протоколом | `ssl://mqtt.cloud.yandex.net` |
| Port | TCP-порт брокера | `8883` |
| Topic | Топик устройства | `$devices/<device-id>/events` |
| Username | MQTT username | по данным вашей учетной записи |
| Password | MQTT password | по данным вашей учетной записи |

### Порядок настройки
1. Укажите `Broker` и `Port`.
2. Укажите `Topic`.
3. Заполните `Username/Password`.
4. Нажмите `Save settings`.
5. На главном экране нажмите `Start`.

**Важно:** ошибка `UnknownHostException` почти всегда означает ошибку в `Broker` (домен или протокол).  
**Примечание:** используйте MQTT endpoint, а не URL веб-консоли.

## 5. Настройка режима уведомлений

В `Settings` есть переключатель:
- `Send notification for every payload (if OFF: only warnings)`

Режимы:
- **ON**: уведомление формируется на каждое входящее сообщение;
- **OFF**: уведомление формируется только при срабатывании порогов.

В payload всегда добавляется служебный флаг:
- `MODE:1` — каждый payload;
- `MODE:0` — только предупреждения.

Это позволяет внешнему обработчику (Cloud Function, Python-сервис, бот) принимать решение о рассылке независимо от мессенджера.

## 6. Работа с главным экраном

Главный экран состоит из двух карточек:

1. **Sensor Data**
   - `Accel X`, `Accel Y`, `Accel Z`;
   - `L (Light)`;
   - `S (Steps)`;
   - `M (Mic peak)`.

2. **MQTT Payload Preview**
   - показывает строку, которая будет отправлена на брокер.

Кнопки:
- `Start` — подключение к MQTT и запуск периодической отправки;
- `Stop` — остановка отправки и отключение;
- `Settings` — переход к конфигурации;
- `About Us` — информационный экран.

### Что означает `N/A`
`N/A` означает:
- датчик отключен в настройках, или
- датчик отсутствует на устройстве, или
- не выдано требуемое разрешение.

## 7. Типовые сценарии использования

### Сценарий A: Базовая эксплуатация
1. Заполните MQTT-настройки.
2. Включите `Accelerometer` и `Light`.
3. Установите интервал `5000 ms`.
4. Нажмите `Save settings`.
5. Нажмите `Start`.
6. Проверьте входящий поток у MQTT-подписчика.

### Сценарий B: Только аварийные уведомления
1. В настройках выключите `Send notification for every payload`.
2. Сохраните.
3. Запустите отправку.
4. Внешний обработчик отправляет уведомления только при превышении порогов.

### Сценарий C: Экономия батареи и трафика
1. Отключите ненужные датчики.
2. Увеличьте интервал отправки (например, 5000–10000 ms).
3. Сохраните.
4. Проверьте, что в payload остаются только активные поля.

### Сценарий D: Диагностика шаго-датчика
1. Убедитесь, что включен `Step counter`.
2. Проверьте разрешение `ACTIVITY_RECOGNITION`.
3. Используйте реальное устройство (не эмулятор).
4. Пройдите несколько шагов и наблюдайте изменение `S`.

## 8. Устранение неполадок

### 8.1 Нет подключения к MQTT
Проверьте:
- корректность `Broker`/`Port`;
- интернет;
- `Username/Password`;
- корректность `Topic`.

Также откройте Logcat и проверьте сообщения `Connection failed`.

### 8.2 Данные не приходят подписчику
Проверьте:
- подписываетесь ли вы на тот же `Topic`;
- есть ли публикации в Logcat (`Published: ...`);
- права подписчика на стороне брокера.

### 8.3 `S (Steps)` всегда `N/A`
Причины:
- нет физического `TYPE_STEP_COUNTER`;
- не выдано разрешение активности;
- датчик выключен в `Settings`.

### 8.4 Ошибка микрофона `start failed`
Причины:
- нет разрешения `RECORD_AUDIO`;
- микрофон занят;
- ограничения эмулятора.

### 8.5 После смены настроек "странное поведение"
1. Нажмите `Stop`.
2. Снова откройте `Settings`, проверьте поля.
3. Нажмите `Save settings`.
4. Нажмите `Start`.
5. При необходимости перезапустите приложение.

## 9. Рекомендации по эксплуатации

- Для стабильной работы шагомера и микрофона используйте физическое устройство.
- Для стресс-тестов не ставьте слишком маленький интервал (например, <1000 ms).
- Для длительного мониторинга используйте отдельный канал анализа/алертов на стороне backend.
- Для аудита инцидентов сохраняйте входящие MQTT-сообщения во внешнем хранилище.

**Важно:** любые пороги предупреждений (`слишком темно`, `слишком шумно`) настраиваются во внешнем обработчике телеметрии и должны быть согласованы с требованиями вашей предметной области.
